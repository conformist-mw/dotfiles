# Network plan — Keenetic + VLAN-сегментация

План наведения порядка в домашней сети (Keenetic Giga + TL-SG108E). Аналог
[`MIGRATION.md`](MIGRATION.md), только про сеть, а не про контейнеры. Справочник «как оно
устроено сейчас» — [`roles/keenetic/README.md`](roles/keenetic/README.md); этот файл —
**поэтапный runbook с решениями**.

## Цель

1. Перестать бояться «настроил и забыл» — всё задокументировано и версионируется.
2. Изолировать видеонаблюдение в отдельный VLAN (не в общей Home-сети).
3. Поднять гостевой Wi-Fi без доступа к локалке.
4. Убрать накопившийся мусор (telnet, стволые DNS, неиспользуемые компоненты).

## Проблема, от которой уходим

- Конфиг роутера жил только в голове и в самом устройстве. WireGuard, dual-WAN, уже
  существующий VLAN2, полусобранный CCTV-ACL — всё это было настроено когда-то и забыто.
- Камеры (NVR + 8 PoE-камер + IpDoor) сидят в **общей Home-сети** `192.168.1.0/24` — любой
  в LAN их видит, и они видят LAN.
- Нет бэкапа конфига роутера в git (есть только бэкап *данных* на флешку — это другое).

## Направление

- **Keenetic не декларативен** (конфиг = список CLI-команд), полноценной реконсиляции как в
  k3s не будет. Поэтому: документация + версионируемый снимок конфига + применение изменений
  осознанными наборами CLI-команд (по SSH или в веб-UI). См. `roles/keenetic/`.
- **Сегментация через 802.1Q**: Keenetic — маршрутизатор/шлюз и DHCP для сегментов,
  TL-SG108E — access-свитч, режет камеры в отдельный VLAN на уровне портов.
- **Изоляция — не «нет маршрута», а firewall**: `isolate-private` уже включён, доступы
  открываются точечными ACL.

### Фаза 1 — камеры в CCTV VLAN

**Решения:** переиспользуем **VLAN2** (уже tagged-транком на порту 3), подсеть
**`192.168.10.0/24`** (под готовый ACL `_WEBADMIN_Home` «home to cctv»), **интернет камерам
разрешён**. Косметика: тег на проводе остаётся `2`, подсеть — `.10` (тег 2 ⇄ 192.168.10.0/24).

**Область:** переезжают только **проводные** камеры за свитчём (NVR sw-p7 + PoE-куст камер
sw-p8 + IpDoor). **Wi-Fi-камеры** (напр. Moes `80:64:7c:c8:f9:61`) остаются в Home — для них
нужен отдельный CCTV-SSID, это отдельная задача (не в этой фазе).

**Moes в NVR (вариант A, применено).** Moes остаётся в Home (`192.168.1.123`), но добавлена в
NVR — а NVR уедет в CCTV и тянет поток сам, т.е. направление CCTV→Home, которое изоляция
режет. Решение — точечная дырка: ACL `_CCTV_NVR_MOES` (`permit ip 192.168.10.112/32 →
192.168.1.123/32`), повешен `ip access-group ... in` на Bridge1. Обратный трафик проходит как
established. Это единственная CCTV→Home дырка; убрать, когда Moes переедет на CCTV-SSID (тогда
NVR увидит её на L2). ⚠ Если веб-UI начнёт «управлять» изоляцией Bridge1, он может перезаписать
единственный inbound access-group на интерфейсе — проверять после правок сегментов в UI.

#### 1a. Keenetic (по SSH, безопасно — на VLAN2 сейчас нет клиентов)

```
interface Bridge1
    description CCTV
    ip address 192.168.10.1 255.255.255.0
!
ip dhcp pool _WEBADMIN_BRIDGE1
    range 192.168.10.50 192.168.10.150
!
```
Резервации проводных камер в новой подсети (сохраняем последний октет для узнаваемости):
```
ip dhcp host 08:a1:89:39:6e:15 192.168.10.112   ! CCTV NVR
ip dhcp host ec:c8:9c:10:5d:d0 192.168.10.113   ! CCTV Front Cam
ip dhcp host ec:c8:9c:10:5d:b9 192.168.10.114   ! CCTV Back Cam
ip dhcp host e8:a0:ed:fc:b8:2e 192.168.10.115   ! CCTV Yard Cam
ip dhcp host e8:a0:ed:55:4d:00 192.168.10.116   ! CCTV Street Cam
ip dhcp host f6:80:00:07:13:49 192.168.10.117   ! CCTV IpDoor
ip dhcp host 50:e5:38:9c:06:ae 192.168.10.118   ! CCTV Garage
ip dhcp host 50:e5:38:9c:06:ab 192.168.10.119   ! CCTV Garden
```
Сохранить: `system configuration save`.
Оставляем как есть: `security-level protected` (изоляция от Home), `ip nat Bridge1`
(интернет), ACL `_WEBADMIN_Home` (Home→CCTV для просмотра; обратное закрыто).

#### 1b. TL-SG108E — 802.1Q VLAN, CCTV = тег 2  (вручную в веб-UI `192.168.1.2`)

| Порт свитча | За ним | VLAN 1 (Home) | VLAN 2 (CCTV) | PVID |
|---|---|---|---|---|
| **p1** | аплинк → Keenetic p3 | untagged | **tagged** | 1 |
| p2 | свитч в зале (downstream) | untagged | — | 1 |
| p3, p4 | свободны | untagged | — | 1 |
| p5, p6 | комната сына (ПК + MikroTik) | untagged | — | 1 |
| **p7** | **NVR** | — | untagged | **2** |
| **p8** | **камерный PoE-свитч** | — | untagged | **2** |

> Аплинк p1 остаётся untagged в VLAN1 (PVID 1) + tagged VLAN2, чтобы Home-трафик и управление
> свитчём (`192.168.1.2`, Management VLAN = 1) не отвалились. easy-smart без API — только руками.
> NVR (p7) и камерный свитч (p8) оба untagged в VLAN2 → они в одном L2-сегменте и видят друг
> друга напрямую, наверх через ACL идёт только просмотр из Home. p2 (зальный свитч) целиком в
> Home — проверить, что за ним нет камер (иначе они не сегментируются этой фазой).

#### 1c. NVR — ⚠ самый болезненный шаг

NVR знает камеры по `192.168.1.x`. После переезда они на `192.168.10.x` → **передобавить
камеры в NVR** (ONVIF-переоткрытие или прописать новые статики). Планировать так, чтобы простой
записи был минимальным.

### Фаза 2 — гостевой Wi-Fi ✅ (сделано)

**Исходное:** точки `GuestWiFi` + `GuestWiFi_5G` были заведены веб-визардом, но остались
сиротами — `down`, без пароля, `security-level private` и **не привязаны ни к какому мосту**;
пул `_WEBADMIN_GUEST_AP` существовал, но пустой (без `range`/`bind`). Визард «Гостевая сеть» в
UI пропал, как только появился ручной CCTV-сегмент — Keenetic считает, что раз лезешь в CLI,
доберёшь сам. Поэтому собрали сегмент руками.

**Решения:**
- Новый сегмент **`Bridge2` «Guest»**, подсеть **`192.168.20.0/24`** (в схему: Home `.1`,
  CCTV `.10`, Guest `.20`), обе точки включены в мост (как Home = Bridge0 с двумя банда́ми).
- SSID **`KeeneticGuest`** / **`KeeneticGuest5`** (зеркалят домашние `Keenetic`/`Keenetic5`),
  WPA2, пароль только на устройстве (в SOPS не нужен, `--sanitize` его режет).
- **`security-level protected`, НЕ `public`** ⚠. Хотели `public` (интернет-only, отрезан от
  всего локального), но **NextDNS не вешается на `public`-сегмент** (`NextDns::Client error:
  unable to set filter for public interface`). Раз нужен фильтр — сегмент `protected`, как CCTV.
- **NextDNS**: отдельный профиль гостя **`9cf78b`** (`nextdns assign interface Bridge2 9cf78b`;
  движок `dns-proxy filter engine nextdns` уже глобальный). Раскладка по сегментам: Home
  `672ede`, Guest `9cf78b`, CCTV `7b6eb3` — у каждого свой уровень фильтрации.
- `ip nat Bridge2` — интернет (NAT пер-сегментный, как `ip nat Home`/`Bridge1`).

**Изоляция (проверено с Android-телефона на гостевой сети):** интернет ✅, NextDNS активен
(my.nextdns.io показал профиль) ✅, Home недоступен (`192.168.1.223`) ✅, **CCTV недоступен**
(`192.168.10.112`) ✅. Последнее — ключевое: гость и CCTV оба `protected`, и Keenetic их всё
равно разделяет (разные сегменты на одном уровне изолированы), так что downgrade `public→
protected` ради NextDNS изоляцию не ослабил. Обе полосы (2.4/5) — один Bridge2, ведут себя
одинаково.

- [x] Точки подняты, WPA2-пароль задан.
- [x] Сегмент изолирован от Home **и от CCTV**, интернет-only + NextDNS-фильтр.
- [x] DHCP-пул `_WEBADMIN_GUEST_AP` дособран (`range .50–.150`, `bind Bridge2`, `enable`).
- [x] `just keenetic-backup` + README.

**Урок (DHCP «не получает IP»):** при первом подключении клиент может писать «не удаётся
получить IP» — первые DHCP DISCOVER не доходят, берёт адрес со второй-третьей попытки. Не паника,
не признак поломки пула. Проверять по `show ip dhcp bindings` (аренда `.20.x` появляется) и
`show associations` (клиент виден на `WifiMaster*/AccessPoint1`).

**Урок (диагностика с того же ноута):** гостевой сегмент `protected` → с него **не достучаться
до роутера `192.168.1.1`** (protected→private режется), т.е. `just keenetic-*` с ноута, сидящего
на гостевой сети, работать не будет. Тестировать изоляцию — с отдельного устройства.

## Отложено (parked)

### keen-pbr — policy-routing доменов через WG до VPS

Идея: часть доменов открывать через WG-туннель до VPS (не поднимая VPN на ноуте). Требует:

- Firmware-компоненты: **добавить** `opkg-kmod-netfilter` + `opkg-kmod-netfilter-addons` (даёт
  ipset/iptables/xtables). `opkg`/`ip6`/`dns-tls`/`dns-https` уже есть.
- Entware-пакеты: `keen-pbr` + `nextdns` (как локальный DoH-прокси).
- **Конфликт с NextDNS:** keen-pbr через `dns-override` ставит свой dnsmasq главным
  резолвером, а сейчас DNS держит штатный `dns-proxy` с NextDNS filter engine. Решение —
  NextDNS переезжает на уровень апстрима keen-pbr: `dnsmasq → 127.0.0.1:5353 (Entware nextdns,
  тот же профиль) → DoH`. Профиль и фильтрация сохраняются, ipset наполняется корректно.
- Outbound = WG-интерфейс `Wireguard0` до VPS.

Причина отложить: не горит; сначала порядок (Фазы 0–3), потом это как отдельный проект.
